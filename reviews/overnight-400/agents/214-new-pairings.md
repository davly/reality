### Tier 1 — bigint + field tower (~1,200 LOC, prerequisite for everything)

**P1 — `pairing/bigint/` (~600 LOC, equivalent to 175-Z1 / 057-T1-BIGINT).** Big-integer arithmetic backed by `math/big.Int` (the FIRST `math/big` import in reality, per 175 §"Precision hazards"). Refs: Bernstein-Yang 2019 *Fast constant-time gcd computation and modular inversion* for constant-time-modular-inverse.

**P2 — `pairing/field/Fp.go` (~240 LOC).** Prime-field `Fp{Modulus *big.Int}` with `Add/Sub/Mul/Square/Inverse/Pow/Sqrt/Legendre/IsZero/Equal`. Specialized for BLS12-381 base prime `p = 4002409555221667393417789825735904156556882819939007885332058136124031650490837864442687629129015664037894272559787` (381 bits, hex `0x1a0111ea397fe69a4b1ba7b6434bae...`), BN254 base prime, BLS12-377 base prime. Montgomery-form and Barrett-form reductions for hot-path. Refs: Montgomery 1985; Barrett 1986; Aranha-Faz-Hernandez-Lopez-Rodriguez 2013 *Faster implementation of scalar multiplication on Koblitz curves*.

**P3 — `pairing/field/Fp2.go` + `Fp6.go` + `Fp12.go` (~360 LOC, equivalent to 175-Z4 / 057-T1-FIELDTOWER).** Quadratic + sextic + cubic-on-quadratic extensions. For BLS12-381: Fp^2 = Fp[u]/(u² + 1), Fp^6 = Fp^2[v]/(v³ - (u+1)), Fp^12 = Fp^6[w]/(w² - v). Sparse-multiplication routines exploiting the line-function shape (`l = (a + bw + cw³)` in Fp^12 has 3 nonzero Fp^2-coefs out of 6; sparse-mul saves ~3× vs full Fp^12-mul). API: full algebra in Fp^12 + cyclotomic-squaring `(a + b·w)² = (a² - b²·v) + (2ab)·w` for elements of cyclotomic-subgroup (used in final-exp hot-path). Refs: Aranha-Karabina-Longa-Gebotys-Lopez 2010; Mrabet-Joye 2017 *Guide to Pairing-Based Cryptography* Ch. 6.

### Tier 2 — Curves G_1 / G_2 + endomorphisms (~1,360 LOC)

**P4 — `pairing/ec/g1.go` BLS12-381 G_1 over Fp (~280 LOC).** Short-Weierstrass `y² = x³ + 4` with Jacobian-projective coordinates (3 Fp values). Add / Double / Negate / ScalarMul / IsOnCurve / IsInfinity / FromBytes / ToBytes. Prime-order subgroup ⊂ E(Fp); cofactor h_1 = `0x396c8c005555e1568c00aaab0000aaab` (76-bit). Subgroup-membership test via Bowe 2019 endomorphism check `[r]P == 0` replaced by `[β]P == [-z²]P` where β is the cubic-root-of-unity GLV-endomorphism (faster: `[β]P` is one Fp-mul + 1 EC-point-copy vs full r-scalar-mul). Cofactor-clearing via Bowe-2019 chosen-cofactor-for-fast-clear. Refs: Bowe 2019 *Faster subgroup checks for BLS12-381*; CIRCL/blst reference impls.

**P5 — `pairing/ec/g2.go` BLS12-381 G_2 over Fp^2 via sextic-twist (~360 LOC).** G_2 is conventionally seen as ⊂ E(Fp^12), but operating there is wasteful; instead represent G_2 = E'(Fp^2) where E' is the sextic twist `y² = x³ + 4·(u+1)` over Fp^2, then map to E(Fp^12) only at line-evaluation time. Twist isomorphism `ψ: E'(Fp^2) → E(Fp^12), (x', y') ↦ (x'·w², y'·w³)`. Cofactor h_2 = a 318-bit composite; cofactor-clearing via Fuentes-Castaneda-Knapp-Rodriguez-Henriquez 2011 endomorphism trick or naive `[h_2]P` for safety. Subgroup check via Scott 2021 *A note on twin pairings*. Refs: Galbraith-Paterson-Smart 2008 §4; Fuentes-Castaneda-Knapp-Rodriguez-Henriquez 2011.

**P6 — `pairing/ec/frobenius.go` Frobenius endomorphism + GLV-GLS scalar split (~320 LOC).** For BLS12-381 G_1: `[β]P = (β·x, y)` where β is a cubic-root-of-unity in Fp; GLV-decomp `k = k_1 + k_2·λ mod r` with λ = -(z² - 1) and `||k_i|| ≤ √r ≈ 2^127`, halving the bit-length of scalars. For G_2 similarly via Galbraith-Lin-Scott. Frobenius-endomorphism on G_2 is `π(P) = (x^p, y^p)`; π² satisfies `π² - z·π + 1 = 0` on G_2 (the curve characteristic polynomial). Used in subgroup-test, cofactor-clearing, and the Miller-loop-final-exp Frobenius-twists. Refs: Gallant-Lambert-Vanstone 2001; Galbraith-Lin-Scott 2009 *Endomorphisms for faster elliptic curve cryptography on a large class of curves*.

**P7 — `pairing/ec/multi_curves.go` curve-parameter tables for BN254/BLS12-377/BLS24-315/BW6-761 (~400 LOC).** Distinct files for each: BN254 `p, r, b, twist-coef, embedding-degree-12, |t-1|=Miller-loop-scalar`; BLS12-377 `u = 0x8508c00000000001, p ≈ 2^377, r ≈ 2^253` (used by Aleo/Zexe; scalar field of BLS12-377 = base field of BW6-761 enabling SNARK-recursion); BLS24-315 (k=24 lower G_2 cost); BW6-761 (Brezing-Weng-on-BLS12-377 outer-circuit recursion curve, k=6 over a 761-bit prime). Each curve gets its own G_1/G_2/Miller-loop/final-exp specialization; ~80 LOC of params per curve + shared template via Go generics over `Curve` interface. Refs: Barreto-Naehrig 2005; Bowe 2017 *BLS12-381 in zk-SNARK applications*; Barreto-Costello-Misoczki-Naehrig-Pereira-Zanon 2019 *Subgroup security in pairing-based cryptography*.

### Tier 3 — Miller loop + final exponentiation (~800 LOC, the cryptographic heart)

**P8 — `pairing/miller/line.go` line and tangent functions (~120 LOC).** For point P ∈ G_1 and Q ∈ G_2 (where Q is in twist-coordinates over Fp^2), `line(P, Q)` evaluates the Fp^12-element representing the line through 2T or T+P (T accumulator) at Q. Sparse-form output: 3 nonzero Fp^2 coefficients out of 6 (the others are forced zero by the line-function structure on the sextic twist). Refs: Miller 1986 *Short programs for functions on curves*; Beuchat-Gonzalez-Diaz-Mitsunari-Okamoto-Rodriguez-Henriquez-Teruya 2010 *High-speed software implementation of the optimal ate pairing over Barreto-Naehrig curves*.

**P9 — `pairing/miller/tate_weil_ate.go` generic Tate / Weil / Ate (~240 LOC).** Tate `t_r(P, Q) = f_{r,P}(Q)^{(q^k-1)/r}`, Weil `w_r(P, Q) = f_{r,P}(Q)/f_{r,Q}(P)` (no final exp but two Miller loops), Ate `a_t(P, Q) = f_{t-1,Q}(P)^{(q^k-1)/r}` (shorter loop t-1 instead of r). Educational completeness — production hot path is P10 only. Refs: Miller 1986; Hess-Smart-Vercauteren 2006 *The Eta pairing revisited*.

**P10 — `pairing/miller/optimal_ate_bls12_381.go` ~280 LOC, KEYSTONE.** Optimal-Ate-pairing for BLS12-381 specifically. Miller-loop scalar = `|u| = 0xd201000000010000` (64-bit, NAF-encoded with sparse Hamming-weight 8 ones — chosen by BLS12-381 designers for fast pairing). Loop body: `(f, T) ← (f² · l_{2T}(P), 2T)`; if u_i = -1: `(f, T) ← (f · l_{T-Q}(P), T-Q)`. Sparse-mul via P3's specialized routine. Final accumulator f ∈ Fp^12 awaiting final-exp. **Constant-time signing variant** uses regular ladder (no NAF skip). **Variable-time verification variant** uses windowed-NAF for ~10% speedup at cost of timing leak (acceptable for verifier). Refs: Vercauteren 2008 *Optimal pairings*; Aranha-Karabina-Longa-Gebotys-Lopez 2010; Beuchat et al 2010.

**P11 — `pairing/finalexp/bls12_381.go` ~180 LOC, KEYSTONE.** Final-exponentiation `f^{(p^12-1)/r}` decomposed via cyclotomic-shortcut: `(p^12-1)/r = (p^6-1) · (p^2+1) · ((p^4 - p^2 + 1)/r)`. Easy part `f^{(p^6-1)·(p^2+1)}` is cheap (one Fp^12-conjugate + one Fp^12-Frobenius-exp + one Fp^12-mul). Hard part `f^{(p^4 - p^2 + 1)/r}` via Aranha-Karabina-Longa-Gebotys-Lopez 2010 ladder: ~13 Fp^12-cyclotomic-squarings + ~10 Fp^12-muls + ~5 Frobenius-exponentiations + scalar-`u` exponentiations. Cyclotomic-squaring is ~50% of the cost of full Fp^12-squaring (Granger-Scott 2010). Refs: Aranha-Karabina-Longa-Gebotys-Lopez 2010 *Faster explicit formulas for computing pairings over ordinary curves*; Granger-Scott 2010 *Faster squaring in the cyclotomic subgroup of sixth degree extensions*.

**P12 — `pairing/finalexp/bn254.go` ~80 LOC.** BN254 final-exp via Devegili-Scott-Dahab 2007 ladder. Different decomp than BLS12-381 (BN254's `(p^12 - 1)/r` factors differently because the curve is non-BLS family). Refs: Devegili-Scott-Dahab 2007 *Implementing cryptographic pairings over Barreto-Naehrig curves*.

### Tier 4 — Hash-to-curve + BLS signatures + IBE (~1,200 LOC)

**P13 — `pairing/hash/htc.go` Wahby-Boneh hash-to-curve RFC 9380 (~400 LOC).** Two-stage construction: first `expand_message_xmd(msg, dst, len_in_bytes)` produces a uniform byte string from SHA-256 (or SHAKE-128) and a Domain-Separation-Tag; second `hash_to_field` reduces bytes to Fp^2 elements; third `map_to_curve` via Simplified-SWU; fourth `iso_map` via 11-isogeny (G_1) or 3-isogeny (G_2) from the SWU-friendly auxiliary curve to BLS12-381. Output: deterministic point in G_1 or G_2 with cofactor cleared. **Gates all of P15-P18 and P21** (BLS-sig requires hash-to-G_1 or G_2; IBE-extract requires hash-to-G_1; Fiat-Shamir requires SHA-based challenge). Hash-to-curve depends on `crypto/sha2` SHA-256 which is currently absent (057 §T1-HASH owns SHA-256). Refs: Wahby-Boneh 2019 *Fast and simple constant-time hashing to the BLS12-381 elliptic curve*; IETF RFC 9380 *Hashing to elliptic curves* (Aug 2023).

**P14 — `pairing/msm/pippenger.go` multi-scalar-multiplication (~360 LOC, equivalent to 175-Z6).** Pippenger bucket-method for `Σ k_i · P_i` over n points in G_1 or G_2. Window-size c ≈ log_2(n) - 2 (e.g., c=14 for n=2^16). GLV-split-aware: each scalar k_i decomposed to (k_{i,1}, k_{i,2}) halving bit-length. Constant-time signing variant (used in BLS-batch-verify with secret scalars) vs variable-time verification variant. Workspace `MSMScratch` allocated once per batch per CLAUDE.md §3. **The single hot-path primitive in KZG.commit, BLS-aggregate-verify, Groth16.commit, PlonK.commit.** Refs: Pippenger 1980 *On the evaluation of powers and monomials*; Bos-Costello-Hisil-Lauter 2014; supranational/blst MSM benchmarks.

**P15 — `pairing/bls/sign.go` BLS-signature ~280 LOC, KEYSTONE.** RFC IRTF-CFRG draft-irtf-cfrg-bls-signature-05. Min-sig-size variant: sk ∈ Fr, pk = [sk]·g_2 ∈ G_2 (96 bytes), sig = [sk]·H(m) ∈ G_1 (48 bytes), verify `e(sig, g_2) == e(H(m), pk)` (one product-of-pairings check via P10+P11). Min-pk-size variant swaps G_1 ↔ G_2. KeyGen: `sk = HKDF-Extract(IKM, salt) mod r` per RFC 5869. Sign / Verify / KeyGen API. Compressed-point serialization (48-byte G_1 / 96-byte G_2 with infinity-flag and y-sign-bit packed in top 2 bits of x-coordinate). Refs: Boneh-Lynn-Shacham 2001 *Short signatures from the Weil pairing*; Boneh-Drijvers-Neven 2018 *Compact multi-signatures for smaller blockchains*; Eth2 spec v1.x.

**P16 — `pairing/bls/aggregate.go` aggregate-BLS + ProofOfPossession (~140 LOC).** Aggregate `agg = Σ sig_i ∈ G_1`. AggregateVerify: `e(agg, g_2) == Π e(H(m_i), pk_i)` (n+1 pairings reduced to 1 product-of-pairings via P10+P11 N-pair extension). ProofOfPossession defends against rogue-key attack: each pk publishes `pop = Sign(pk-bytes, sk)` so a verifier checks `e(pop, g_2) == e(H_pop(pk-bytes), pk)`. Without PoP, an attacker can produce `pk' = pk_attacker - Σ pk_i` and forge an aggregate-sig as `[sk_attacker]H(m)` valid against the honest set. Refs: Boldyreva 2003; Ristenpart-Yilek 2007; Boneh-Drijvers-Neven 2018.

**P17 — `pairing/bls/threshold.go` threshold-BLS via Shamir (~120 LOC).** Distribute sk via Shamir-(t,n)-secret-sharing in Fr; each party signs with sk_i to produce sig_i = [sk_i]·H(m); any t signatures combine via Lagrange-interpolation in the exponent: `sig = Σ λ_i · sig_i ∈ G_1` where λ_i = Π_{j≠i} (j / (j-i)). Verifies as a normal BLS-sig under the master pk. **The single most-deployed threshold-signature in production blockchain** (Eth2 Distributed-Validator-Tech / Obol / SSV / Ethereum Trusted Execution Environments). Refs: Boldyreva 2003 §4; Stinson-Strobl 2001.

**P18 — `pairing/bls/batch_verify.go` batch verification (~120 LOC).** For n distinct messages and signatures, batch-verify checks `e(Σ r_i · sig_i, g_2) == Π e(H(m_i), [r_i] pk_i)` where r_i are random scalars (Camenisch-Hohenberger-Pedersen 2007 random-linear-combination). Reduces n+1 pairings to 2 pairings (via product-of-pairings). r_i drawn from a 128-bit space gives 128-bit forgery-resistance via Schwartz-Zippel. Refs: Camenisch-Hohenberger-Pedersen 2007; Bowe-Grigg-Hopwood 2017 §6.

**P19 — `pairing/ibe/boneh_franklin.go` Boneh-Franklin IBE (~280 LOC).** Identity-Based-Encryption (Boneh-Franklin 2001, the prototypical pairing-crypto application). Setup: master sk = s ∈ Fr, master pk = [s]·g_1. Extract(ID): output user-sk = [s]·H_1(ID) ∈ G_1 where H_1 hashes-to-G_1 (P13). Encrypt(ID, m): random r ∈ Fr, output (c_1 = [r]g_1, c_2 = m XOR H_2(e(H_1(ID), [s]g_1)^r)). Decrypt(user-sk, (c_1, c_2)): m = c_2 XOR H_2(e(user-sk, c_1)). Type-3-pairing variant (asymmetric, modern security). Refs: Boneh-Franklin 2001 *Identity-based encryption from the Weil pairing*; Galindo 2005 *Boneh-Franklin identity-based encryption revisited*.

### Tier 5 — KZG + Groth16 + PlonK (~1,640 LOC)

**P20 — `pairing/kzg/srs.go` + `commit.go` + `open.go` + `verify.go` (~400 LOC, equivalent to 175-Z14).** Univariate KZG. SRS = (`[1]g_1, [τ]g_1, [τ²]g_1, ..., [τ^{d-1}]g_1`, `[1]g_2, [τ]g_2`) for max-degree d. Commit(f) = `[f(τ)]_1 = Σ f_i · [τ^i]_1` (one P14-MSM). Open(f, z): compute quotient `q(X) = (f(X) - f(z)) / (X - z)` via long-division; π = `[q(τ)]_1`. Verify(c, z, y, π): `e(c - [y]_1, [1]_2) == e(π, [τ]_2 - [z]_2)` (one product-of-pairings via P10+P11). BatchOpen via random-linear-combination. MultiPoint Feist-Khovratovich 2020 for opening at n distinct points in O(d log d) instead of O(n·d). EIP-4844 wire-format adjacent. **The single math-engine of PlonK/Sonic/Marlin/Caulk verifiers.** Refs: Kate-Zaverucha-Goldberg 2010 *Constant-size commitments to polynomials and their applications*; Feist-Khovratovich 2020 *Fast amortized KZG proofs*.

**P21 — `pairing/groth16/{setup,prove,verify}.go` ~520 LOC.** Groth16 SNARK over R1CS. Per-circuit-trusted-setup: given R1CS matrices (A, B, C) of size m×n with witness w, generate CRS = `({[A_i(τ)]_1}, {[B_i(τ)]_1}, {[B_i(τ)]_2}, {[C_i(τ)]_1}, {[h_i(τ)·t(τ)/δ]_1})` (~12 MSMs over the lagrange-basis-encoded R1CS). Prove: 3 G_1 + 1 G_2 elements (`A = [A(τ) + r·δ]_1, B = [B(τ) + s·δ]_2, C = [C-related polynomial]_1`). Verify: 3 pairings (one is a product-of-2-pairings against the public-input commitment). **128-byte proofs for BN254, 192-byte for BLS12-381 — the smallest known SNARK proofs in production.** Soundness via knowledge-of-coefficient assumption (q-DLOG in algebraic-group-model, Fuchsbauer-Kiltz-Loss 2018). Refs: Groth 2016 *On the size of pairing-based non-interactive arguments*; Bowe-Gabizon-Miers 2017 *Scalable multi-party computation for zk-SNARK parameters in the random-beacon model* for ceremony.

**P22 — `pairing/plonk/{indexer,prover,verifier}.go` ~720 LOC.** PlonK over a universal SRS (KZG-based). Indexer: takes a circuit (encoded as 5 selector polynomials q_L, q_R, q_O, q_M, q_C plus copy-permutation polynomial S_σ) and outputs the verification-key (commitments to selectors + permutation). Prover: 5-round protocol — round 1 commits to wire-witness polynomials a, b, c; round 2 commits to permutation-product polynomial Z; round 3 commits to quotient-polynomial t split into 3 chunks; round 4 evaluates a, b, c, S_σ, Z at challenge ζ; round 5 batches the openings into one KZG-opening at ζ + one at ζ·g (where g is a generator of the multiplicative subgroup of size n). Verifier: 1 pairing (after batching). **Universal-SRS + per-circuit-VK + 9-element-proof — the modern industry standard.** Used by Aztec/Aleo/Polygon-zkEVM/Mina. Refs: Gabizon-Williamson-Ciobotaru 2019 *PlonK*; gnark-crypto reference impl.

### Tier 6 — Lookup arguments frontier (~620 LOC, 2024-2026 research)

**P23 — `pairing/plookup/plookup.go` ~140 LOC.** Plookup (Gabizon-Williamson 2020) lookup argument: given table T and query trace W, prove W ⊆ T via grand-product of `Z(X) = Π (1 + β·(W_i || sorted(T,W)_i)) / (1 + β·(T_i || ...))`. Plugs into PlonK via one extra commitment. Refs: Gabizon-Williamson 2020 *plookup: A simplified polynomial protocol for lookup tables*.

**P24 — `pairing/caulk/caulk.go` + `caulk_plus.go` ~200 LOC.** Caulk + Caulk+ (Zapico-Buterin-Khovratovich-Maller-Nitulescu-Mukhopadhyay 2022) cached-quotient-lookups: given a precomputed table-commitment `[T(τ)]_1`, a query of t lookups costs `O(t·log t)` field-ops + `O(t)` G_1-points (sublinear in |T|). Refs: Zapico-Buterin-Khovratovich-Maller-Nitulescu-Mukhopadhyay 2022; Pedrouzo-Ulloa et al 2023.

**P25 — `pairing/cq/cq.go` cq-protocol ~140 LOC.** cq (Eagen-Fiore-Gabizon 2022) further-improved cached-quotient via FK-amortized openings — log-factor-improvement over Caulk+. Refs: Eagen-Fiore-Gabizon 2022 *cq: Cached quotients for fast lookups*.

**P26 — `pairing/sonic/sonic.go` Sonic ~140 LOC.** Sonic (Maller-Bowe-Kohlweiss-Meiklejohn 2019) — universal-SRS predecessor to PlonK. Useful for educational completeness; no new production deployments since PlonK (2019) shipped. Refs: Maller-Bowe-Kohlweiss-Meiklejohn 2019 *Sonic: Zero-knowledge SNARKs from linear-size universal and updateable structured reference strings*.

### Tier 7 — Cryptanalytic awareness (~600 LOC)

**P27 — `pairing/security/exTNFS.go` ~200 LOC.** Extended-Tower-Number-Field-Sieve (Kim-Barbulescu 2016) attack on pairing-friendly fields: for composite embedding-degree k (like k=12=2²·3), the discrete-log-in-Fp^k cost drops from `L_p^k(1/3, ...)` to `L_p^k(1/3, c')` with c' < c, effectively halving the "pairing security level" of BN254 from claimed 128-bit to actual ~100-bit. **This is why Eth2 migrated from BN254 to BLS12-381 in 2018.** P27 documents attack complexity, current security estimates per curve (BN254: ~100-bit, BLS12-381: ~120-bit, BLS12-377: ~120-bit, BLS24-315: ~127-bit, BW6-761: ~125-bit), and provides a `SecurityLevel(curve)` API consumers can branch on. Refs: Kim-Barbulescu 2016 *Extended tower number field sieve*; Menezes-Sarkar-Singh 2016 *Challenges with assessing the impact of NFS advances on the security of pairing-based cryptography*.

**P28 — `pairing/security/frey_ruck.go` ~80 LOC.** Pairing-inversion (Frey-Rück 1994) hardness commentary: DLP in G_T `→` DLP in G_1 is reducible (the easy direction); reverse reduction "given e(P, Q) compute Q given P" is the **pairing-inversion problem** which is open and conjectured hard — every pairing-crypto-hardness assumption (BDH, q-SDH, BDHE, etc.) implicitly relies on pairing-inversion-hardness. **Documentation primitive only**, not executable; complements P27. Refs: Frey-Rück 1994 *A remark concerning m-divisibility and the discrete logarithm in the divisor class group of curves*; Galbraith-McKee-Valenca 2008 *Pairing inversion is harder than DLP in G_T*.

## 4. Precision hazards specific to pairing-crypto

- **Constant-time discipline.** Miller-loop signing (BLS-sign with secret point [sk]·H(m)) MUST be constant-time on the scalar — no NAF-skip-of-zero-bits, no windowed-addition-table-lookup-with-secret-index, no Montgomery-inverse-via-Fermat-with-input-dependent-loop-count. Reality v0.10.0 has zero constant-time discipline because it ships zero crypto today; introducing it via P15 is the **first time reality faces side-channel-resistance** as a design constraint. Recommend a `Curve.SignSecure()` API that returns guaranteed-constant-time vs `Curve.VerifyFast()` that allows variable-time.
- **GLV-decomposition carries a security trap.** GLV-split scalar k = k_1 + k_2·λ with `||k_i|| ≤ √r` requires lattice reduction over the ring `Z[λ]`; the reduction must be done in the SETUP phase (compile-time constants for short-basis vectors). A naive runtime GLV reduction leaks bits of k via timing. Reality must hardcode the BLS12-381 short-basis-vectors as compile-time tables.
- **Hash-to-curve constant-time-modular-square-root.** Wahby-Boneh hash-to-curve uses the `is_square(x)` test in Simplified-SWU; this is the Tonelli-Shanks algorithm whose execution time depends on the input. Reality must use the constant-time variant (Bernstein-Yang 2019 *Fast constant-time gcd computation and modular inversion*).
- **Subgroup-membership-test forgery.** A point on E(Fp) (or E'(Fp²)) but NOT in the prime-order subgroup G_1 (or G_2) leads to small-subgroup attacks on BLS-sig; verify must check subgroup membership BEFORE pairing. The naive `[r]·P == 0` check is one full scalar-mul (O(log r)) which is wasteful; Bowe 2019 endomorphism-check `[β]·P == [-z²]·P` runs in 3 Fp-muls + 1 EC-add. The cheaper check is mandatory; reality must NOT ship the naive check even as fallback.
- **Multi-pairing batching arithmetic is associative-not-commutative-in-Fp¹².** `Π e(P_i, Q_i)` accumulates products in Fp^12 inside the Miller loop AFTER each iteration's line-evaluation; the accumulator order matters for cyclotomic-shortcut applicability (consecutive multiplications must stay in cyclotomic subgroup). A naive batch-pairing that sums G_1-points first (`Σ P_i`) breaks if the sum is non-cyclotomic; the correct batching is `Π e(P_i, Q_i)` where each pair is independently Miller-looped and the products are accumulated in Fp¹². Reality's BatchPair API must enforce this.
- **Trusted-setup-ceremony disclaimer.** KZG / Groth16 / PlonK / Sonic all require a trusted-setup output (Powers-of-Tau ceremony for KZG / per-circuit Groth16 ceremony / universal-PlonK ceremony). Reality does NOT generate the ceremony output (175 §"OUT" boundary, this slot inherits); reality CONSUMES caller-supplied SRS bytes. The package docstring MUST declare this OUT boundary loud — a consumer running `pairing.NewKZGSRS(maxDegree)` with reality-only-input will get a deterministic-but-INSECURE SRS (since reality's PRNG is deterministic per 175 §"Precision hazards"). Real SRS bytes must be loaded from an independently-verified ceremony output (Filecoin / Zcash / Mainnet KZG / EIP-4844 Trusted-Setup-Ceremony).
