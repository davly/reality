## 1. The twenty primitives — IF you were to ship this

Tier numbering: T1 = elliptic-curve + j-invariant foundation, T2 = isogeny morphism + Vélu, T3 = isogeny graph walks, T4 = quaternion algebra + Deuring, T5 = CSIDH class-group action, T6 = SQISign + SQISign-HD, T7 = cryptanalysis pedagogy.

### Tier 1 — Elliptic curves + j-invariant (~720 LOC, prerequisite)

**I1 — `isogeny/curve/weierstrass.go` ~180 LOC.** `Curve{A, B Field; Field Fp}` representing E: y² = x³ + Ax + B over Fp or Fp2. Methods: `IsOnCurve(P Point) bool`, `Discriminant() Field` (Δ = −16(4A³ + 27B²)), `IsSingular() bool` (Δ = 0), `JInvariant() Field` (j = −1728·(4A)³/Δ — note FIPS-186 uses 1728·4A³/(4A³+27B²); the two conventions differ by sign and `reality` must pin one in golden vectors). **Cross-link to 057 T1-EC:** 057 ships the same curve type for ECDSA/EdDSA over Fp; I1 SUPERSETS that with Fp2 extension (required because supersingular curves over Fp are rare — most supersingular j-invariants live in Fp2). Refs: Silverman *AEC* III.1; Costello *Computational Aspects of Isogeny-Based Cryptography* §2.

**I2 — `isogeny/curve/jinvariant.go` ~80 LOC.** Standalone j-invariant computation + isomorphism testing. Two curves E_1, E_2 are F̄p-isomorphic iff j(E_1) = j(E_2). Method: `IsIsomorphic(E1, E2 Curve) bool`, `IsomorphismMap(E1, E2 Curve) (Map, error)` returning the affine isomorphism (x, y) ↦ (u²x + r, u³y + u²sx + t) when one exists. Refs: Silverman *AEC* III.1.4.

**I3 — `isogeny/curve/supersingular.go` ~140 LOC — KEYSTONE.** Supersingular-vs-ordinary discriminator. Three algorithms:
- **Naive** (slow): compute #E(Fp) via Schoof's algorithm or BSGS, check if t = p+1 − #E ≡ 0 mod p.
- **Sutherland 2012** (fast): walk a short isogeny chain, compute the trace of Frobenius via Hilbert class polynomial. ~500 LOC if shipped fully; recommend stub.
- **Cheap heuristic** (Atkin-Morain 1993): for small p, look up j(E) in a precomputed table of supersingular j-invariants mod p (~p/12 of them). Useful for pedagogy.

Used by every isogeny scheme to validate that public-key curves are actually supersingular. Refs: Sutherland 2012 *Identifying supersingular elliptic curves*; Atkin-Morain 1993.

**I4 — `isogeny/curve/montgomery.go` ~140 LOC.** Montgomery-form curve `By² = x³ + Ax² + x` for x-only point arithmetic. Differential addition: `xADD((x_P, x_Q, x_{P-Q})) → x_{P+Q}` in 4M+2S. Doubling: `xDBL(x_P, A24) → x_{2P}` in 4M+2S where A24 = (A+2)/4 precomputed. **The canonical curve model for SIDH/CSIDH/SQISign because x-only arithmetic eliminates the y-coordinate sign ambiguity that complicates isogeny computation.** Cross-link 057-T1-EC: 057's Curve25519 ladder uses the same Montgomery form, but for X25519 not isogenies. Refs: Costello-Hisil 2017 *A simple and compact algorithm for SIDH with arbitrary degree isogenies*; Costello *Pairings for Beginners* §1.

**I5 — `isogeny/curve/edwards.go` ~120 LOC.** Twisted-Edwards form for completeness — supersingular Edwards curves exist, used in some research papers but NOT in mainstream isogeny crypto. Recommend STUB only — most isogeny work happens on Montgomery curves, Edwards is rare in this domain. Skip in production.

**I6 — `isogeny/field/fp2.go` ~140 LOC.** Quadratic extension Fp2 = Fp[i]/(i² + 1) (or i² − non-residue depending on p mod 4). Add/Sub/Mul/Square/Inverse/Conjugate/Frobenius. **Required because most supersingular j-invariants live in Fp2, not Fp.** Cross-link 057-T1-FIELDTOWER: same construction, but isogeny crypto stops at Fp2 (does NOT need Fp4 / Fp6 / Fp12 like pairings do). Refs: Costello-Longa-Naehrig 2016 *Efficient algorithms for supersingular isogeny Diffie-Hellman* §3.

### Tier 2 — Isogeny morphism + Vélu (~880 LOC)

**I7 — `isogeny/iso/morphism.go` ~120 LOC.** `Isogeny{Source, Target Curve; Degree int; Kernel Point; Map RationalMap}` representing φ: E → E'. Methods: `Apply(P Point) Point`, `Dual() Isogeny` (returns φ̂ with deg(φ̂) = deg(φ) and φ̂ ∘ φ = [deg φ]), `Compose(other Isogeny) Isogeny`, `IsSeparable() bool`. Composition of isogenies — the building block for isogeny chains in SIDH/SIKE/CSIDH. Refs: Silverman *AEC* III.4.

**I8 — `isogeny/iso/velu.go` ~200 LOC — KEYSTONE.** Vélu 1971 explicit formulas. Given a curve E and a finite subgroup H ⊂ E generated by a point P of odd prime order ℓ, compute the rational map φ: E → E/H of degree ℓ. **Per-coordinate output:** φ_x(x) = x + Σ_{Q ∈ H \ {O}} (x − x_Q)^{−1} · ((numerator from V_Q, U_Q))^{−1}; symbolic implementation requires ((ℓ−1)/2) field operations per coordinate. Naive Vélu has Õ(ℓ) complexity per isogeny — fine for small ℓ (≤ 50), prohibitive for large ℓ. Refs: Vélu 1971 *Isogénies entre courbes elliptiques*; Costello *Pairings for Beginners*.

**I9 — `isogeny/iso/velu_xonly.go` ~160 LOC.** Costello-Hisil 2017 x-only Vélu for Montgomery curves. Eliminates y-coordinate computation, halves the operation count, AND avoids the y-sign ambiguity. Used by all SIDH/CSIDH/SQISign reference implementations. Output: the Montgomery image curve coefficient A' and the x-coordinate map. Refs: Costello-Hisil 2017.

**I10 — `isogeny/iso/sqrtvelu.go` ~280 LOC — KEYSTONE.** Bernstein-De Feo-Leroux-Smith 2020 √élu (square-root Vélu). Reduces per-isogeny complexity from Õ(ℓ) to Õ(√ℓ) by reformulating the Vélu sum as a polynomial-multipoint-evaluation problem solved via FFT-style baby-step-giant-step over the kernel. **THE single most important advance in isogeny computation post-2020 — enables B-SIDH (Costello 2020), pSIDH, FESTA, SQISign-HD with large primes ℓ ~ 2^40+ that were previously infeasible with naive Vélu.** Implementation requires fast polynomial arithmetic over Fp (cross-link to 211 NTT family — but isogeny-Vélu uses LARGE-prime NTT not small-prime negacyclic, different parameter regime). Refs: Bernstein-De Feo-Leroux-Smith 2020 *Faster computation of isogenies of large prime degree*.

**I11 — `isogeny/iso/chains.go` ~120 LOC.** Isogeny-chain composition. Given a sequence of small-degree isogenies φ_i: E_i → E_{i+1}, walk them sequentially. **2-isogeny chains** (used in SIDH/SIKE for Alice's curve walk via 2-isogenies): height a chain of length ~370 for SIKEp751. **3-isogeny chains** (Bob's curve walk for SIKEp751): length ~239. **Optimal-strategy traversal** (De Feo-Jao-Plût 2014): pre-compute the strategy tree that minimizes total isogeny + scalar-multiplication operations; this is a discrete-optimization problem (knapsack-like) with closed-form solutions tabulated in the SIKE reference. Refs: De Feo-Jao-Plût 2014; Costello-Longa-Naehrig 2016.

### Tier 3 — Isogeny graph + walks (~480 LOC)

**I12 — `isogeny/graph/pizer.go` ~180 LOC.** Pizer 1980 supersingular isogeny graph G(p, ℓ). Vertex set = {supersingular j-invariants over Fp²} (size ≈ p/12). Edge set = ℓ-isogenies. (ℓ+1)-regular Ramanujan graph (eigenvalue bound 2√ℓ from second-largest-eigenvalue). Random-walk mixing time O(log p). **Implementation:** start at j_0, at each step compute all (ℓ+1) neighbors via Vélu on each ℓ-torsion subgroup, choose one uniformly at random. Cross-link to `graph/` package: graph algorithms (BFS/DFS/random-walk) overlap, but vertex-set is dynamic (computed from j-invariants on-the-fly, not pre-enumerated) so reuse is partial. Refs: Pizer 1980 *An algorithm for computing modular forms on Γ_0(N)*.

**I13 — `isogeny/graph/walk.go` ~140 LOC.** Random walk on Pizer graph for hash-function and key-generation. Boneh-Kogan-Woo 2020 isogeny-OPRF construction. Walk length Θ(log p) ≈ 500 steps for 256-bit security. Output: the j-invariant after walking. **Used as the hash function in CGL hash (Charles-Goren-Lauter 2009) — the FIRST isogeny-based cryptographic primitive.** Refs: Charles-Goren-Lauter 2009 *Cryptographic hash functions from expander graphs*.

**I14 — `isogeny/graph/ramanujan.go` ~160 LOC.** Spectral analysis of the Pizer graph. Compute the second-largest eigenvalue (theory says ≤ 2√ℓ; verify experimentally). **Useful for security-parameter selection** — the isogeny-walk hash function's security depends on Ramanujan-quality mixing. Cross-link to `linalg/` eigenvalue solvers. Refs: Lubotzky-Phillips-Sarnak 1988 *Ramanujan graphs*; Pizer 1990.

### Tier 4 — Quaternion algebra + Deuring (~860 LOC)

**I15 — `isogeny/quat/algebra.go` ~200 LOC.** Quaternion algebra B = (a, b / Q) with basis {1, i, j, ij} satisfying i² = a, j² = b, ij = −ji. **Specifically B_{p,∞} = (−q, −p / Q)** for the smallest q such that the resulting algebra ramifies at exactly {p, ∞}. Add/Sub/Mul/Conjugate/Norm/Trace/Reduced-trace/Reduced-norm. Note: this is OVER Q — needs **rational-number arithmetic** (numerator/denominator pairs as big-integers), which `crypto/` doesn't ship at all. **Mathematically UNRELATED to `geometry/quaternion.go`** (which is 3D rotations over R). Refs: Voight 2021 *Quaternion algebras* (textbook).

**I16 — `isogeny/quat/order.go` ~280 LOC.** Maximal orders + Eichler orders in B_{p,∞}. `Order{Basis [4]Quat; Discriminant *big.Int}`. Standard maximal order O_0 ⊂ B_{p,∞}, ideals I ⊂ O_0 (left/right/two-sided), ideal multiplication, ideal norm, ideal class group representatives. **THE single hardest math primitive in isogeny crypto** — equivalent to Magma's `QuaternionOrder` / PARI/GP's `alginit` machinery, both of which are 5,000+ LOC. Reality from-scratch impl would be ~1,500 LOC at production quality; recommend STUB at 280 LOC covering only the operations needed for SQISign basic verification. Refs: Voight 2021; KLPT algorithm (Kohel-Lauter-Petit-Tignol 2014) for ideal-norm-equation-solving.

**I17 — `isogeny/quat/deuring.go` ~380 LOC.** Deuring correspondence as a computational object. `DeuringMap` providing: given a supersingular curve E, output a Z-basis of End(E) as quaternions in O_0 ⊂ B_{p,∞}; given a left ideal I ⊂ O_0, output the corresponding isogeny φ: E → E/I; given an isogeny chain, output the composed-ideal. **The Eisenträger-Hallgren-Lauter-Morrison-Petit 2018 algorithm** for End(E) computation runs in time exp(O((log p)^{1/2})) classical — sub-exponential but NOT polynomial. **Wesolowski 2022** proved this is equivalent to the Endomorphism Ring Problem which underpins SQISign's security. Refs: Eisenträger-Hallgren-Lauter-Morrison-Petit 2018 *Supersingular isogeny graphs and endomorphism rings: reductions and solutions*; Wesolowski 2022 *The supersingular isogeny path and endomorphism ring problems are equivalent*.

### Tier 5 — CSIDH class-group action (~640 LOC)

**I18 — `isogeny/csidh/csidh.go` ~280 LOC — POST-CASTRYCK-DECRU SURVIVOR.** CSIDH = Castryck-Lange-Martindale-Panny-Renes 2018. Class group Cl(O) of imaginary-quadratic order O acts on supersingular curves over Fp (p ≡ 3 mod 4, p = 4·ℓ_1·...·ℓ_n − 1 with ℓ_i small odd primes). Public key: a curve E_a = [a]·E_0 where a ∈ Cl(O). Diffie-Hellman: Alice and Bob each compute [a]·E_0 and [b]·E_0, exchange, both compute [ab]·E_0 = [a]·[b]·E_0 (commutativity from Cl(O) being abelian — KEY DIFFERENCE from SIDH which is NOT commutative). **Still secure post-Castryck-Decru** because the attack required torsion-point images, which CSIDH does not reveal. Performance: CSIDH-512 ~80ms key exchange (slow vs ECDH ~0.1ms), CSIDH-1024 ~600ms. Refs: Castryck-Lange-Martindale-Panny-Renes 2018 *CSIDH: An efficient post-quantum commutative group action*.

**I19 — `isogeny/csidh/csi_fish.go` ~200 LOC.** CSI-FiSh signature scheme (Beullens-Kleinjung-Vercauteren 2019). Fiat-Shamir wrapper around CSIDH's group action, ~263 byte signatures. Precomputation requirement: structure of the class group Cl(O) must be determined (~2^32 evaluations for CSIDH-512), so this is not a plug-and-play scheme — only CSIDH-512 has full structure, larger primes require more precomputation. Refs: Beullens-Kleinjung-Vercauteren 2019 *CSI-FiSh: Efficient isogeny based signatures through class group computations*.

**I20 — `isogeny/csidh/oprf.go` ~160 LOC.** Boneh-Kogan-Woo 2020 OPRF from CSIDH. Two-party-protocol where the server applies a private class-group element to a client-supplied curve. Used in private-set-intersection, password-authenticated-key-exchange. Active 2024 research, no production deployment. Refs: Boneh-Kogan-Woo 2020 *Oblivious pseudorandom functions from isogenies*.

### Tier 6 — SQISign + SQISign-HD (~1,420 LOC, deferred)

**I21 — `isogeny/sqisign/sqisign.go` ~600 LOC.** SQISign 2020 (De Feo-Kohel-Leroux-Petit-Wesolowski). Identification protocol: prover knows End(E_pk) where E_pk is a supersingular curve; challenge = a random isogeny φ_chal: E_pk → E_chal of small degree; response = an isogeny φ_resp: E_pk → E_chal computed via the prover's knowledge of End(E_pk) using the KLPT algorithm. ## 2. The candor section: SHOULD reality ship this?

**Recommendation: NO.** Five concrete reasons:

1. **Catastrophic-attack history.** SIDH/SIKE was a NIST PQC Round-4 candidate with substantial industry backing (Microsoft Research had production-ready C implementations). It was broken in **<1 hour** by a 7-page paper using genus-2 isogeny machinery (Castryck-Decru 2022). Maino-Martindale and Robert independently published variants the same month. The breakage was so severe and so unexpected that the NIST PQC effort had to delete an entire family of candidates from consideration. Any isogeny scheme reality ships today (CSIDH, SQISign, SQISign-HD, FESTA, pSIDH, QFESTA) faces the same risk profile — these are **research-grade primitives** with active 2024–2026 cryptanalysis literature.

2. **No standardization pathway.** Lattice cryptography has FIPS-203 (ML-KEM), FIPS-204 (ML-DSA), and FIPS-206 (FN-DSA) — three NIST standards with KAT vectors, ACVP test suites, and clear interoperability requirements. Hash-based has FIPS-205 (SLH-DSA) + RFC 8391 (XMSS) + RFC 8554 (LMS). Isogeny crypto has **zero NIST standards** post-SIDH-collapse. SQISign was a NIST Round-4 onramp candidate but was NOT selected for standardization; the next NIST onramp (initiated 2023) added SQISign-HD as a candidate but with a 5+ year evaluation timeline — meaning earliest possible standardization is 2029.

3. **Cross-language golden-file mandate (CLAUDE.md §1) is impossible.** Reality's design contract requires bytewise-identical output across Go/Python/C++/C# substrates. For lattice crypto this is achievable because FIPS-203/204 specify the exact byte encoding of every polynomial, key, and ciphertext. For isogeny crypto, **there is no canonical byte encoding** of "an isogeny" or "a supersingular curve" — the SQISign reference, FESTA reference, SQISign-HD reference, and CSIDH reference each use different conventions (different curve models, different field representations, different isogeny-chain encodings). Without a canonical encoding, cross-language golden vectors are not even definable.

4. **Mathematical specificity is extreme.** Reality's other 22 packages cover dual-use math: linear algebra serves graphics + simulation + ML + crypto + ZK; FFT serves signal processing + lattice crypto + polynomial arithmetic; quaternions (the geometry kind) serve robotics + orbital mechanics + 3D animation. **Isogeny crypto's mathematical substrate (supersingular elliptic curves, quaternion algebras over Q, Eichler orders, Deuring correspondence, theta coordinates) has ZERO non-cryptographic consumers in reality's plausible roadmap.** It would be the only single-purpose math package.

5. **Implementation complexity is PhD-level.** Lattice crypto can be implemented from FIPS-203/204 by a senior crypto engineer in ~6 weeks (the spec is ~80 pages and self-contained). Isogeny crypto requires understanding Silverman's *Arithmetic of Elliptic Curves* (~500 pages of algebraic geometry), Voight's *Quaternion Algebras* (~700 pages), the De Feo-Kohel-Leroux-Petit-Wesolowski 2020 paper (32 pages dense), AND the Robert 2024 theta-coordinate machinery (active research). Approximately **50 people in the world** have implemented isogeny crypto to production quality. Asking reality to ship this is asking it to enter a domain where there are essentially no off-the-shelf reference implementations to validate against.

**What to ship instead:** A 200-LOC `isogeny/INTRO.md` document explaining the field, citing the canonical references (Silverman, De Feo lecture notes, SQISign-HD spec), and stating reality's deliberate non-coverage of this area. If a consumer materializes (e.g., a future zk-SNARK ecosystem chooses an isogeny-based proof system, or a privacy-protocol vendor needs CSIDH-OPRF), revisit at that time — but for v1.x, the right answer is **"reality does not ship isogeny cryptography by design."**

If reality MUST ship something isogeny-flavored as a defensive minimum (e.g., for academic credibility / pedagogy / future-optionality), the cheapest defensible target is **I1 + I2 + I3 = elliptic-curve-over-Fp + j-invariant + supersingular-detector ≈ 280 LOC**. Document it as **"pedagogical primitive, NOT a cryptosystem"** and stop there.
